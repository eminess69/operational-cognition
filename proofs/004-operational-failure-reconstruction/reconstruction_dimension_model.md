# Reconstruction Dimension Model

The dimensions below define what an operational failure reconstruction audit may need to recover from public artifacts. Each row is preliminary and subject to evidence validation.

Allowed claim levels: `observed`, `inferred`, `hypothesis`, `unresolved`.

| Dimension | Definition | Why It Matters | Public Evidence Surfaces | Reconstruction Limitations | Unresolved Questions | Current Claim Level |
| --- | --- | --- | --- | --- | --- | --- |
| transcript reconstruction | Reconstructing visible user, assistant, system-visible summaries, and successor transcript boundaries in order. | Transcript state provides the visible task narrative and can reveal when user intent, constraints, or summaries changed. | OpenClaw trajectory and compaction documentation; OpenHands conversation and condenser documentation. | Transcript continuity may diverge from model-visible context after compaction, condensation, pruning, or restart. | Are omitted ranges, successor transcript boundaries, and summary insertion points preserved with stable references? | observed |
| event reconstruction | Reconstructing typed events, action/observation pairs, session state updates, and event ordering. | Failure pathways often depend on event transitions rather than only natural-language transcript text. | OpenHands event and conversation architecture; OpenClaw trajectory bundles. | Event schemas may not by themselves capture environment state, memory state, full tool result content, or runtime configuration. | Which event IDs, event payloads, and state transitions remain available in public replay artifacts? | observed |
| tool/action reconstruction | Reconstructing tool calls, command actions, browser actions, observations, errors, truncation, and call/result pairing. | Tool/action lineage is necessary to explain operational behavior and identify where a pathway diverged. | OpenClaw trajectory bundles; OpenHands events; OpenHands browser session recording. | Public docs may describe action surfaces without proving complete result retention, redaction boundaries, or schema-version retention. | Are tool schemas, result hashes, redaction reasons, and error payloads retained with enough provenance? | inferred |
| memory reconstruction | Reconstructing durable memories, recall queries, recall results, memory flushes, indexes, and cross-session memory surfaces. | Memory can change later action selection and can bridge multiple sessions or compressed contexts. | OpenClaw memory and compaction documentation; OpenHands condenser documentation where history compression affects model-visible context. | Memory indexes, recalled snippets, and memory write authority may not be visible from replay alone. | Can public artifacts distinguish stored memory, recalled memory, summarized memory, and absent memory? | unresolved |
| environment reconstruction | Reconstructing workspace, filesystem, sandbox, browser, runtime, container, dependency, network, and external-service state references. | Agents fail or recover based on state outside the transcript, including files, processes, permissions, and runtime configuration. | OpenHands agent server and conversation architecture; OpenClaw trajectory bundle metadata. | Public traces may omit complete filesystem diffs, dependency versions, container images, credentials boundaries, or external state snapshots. | Which environment refs or hashes are required for a replayable causal explanation? | unresolved |
| temporal reconstruction | Reconstructing timestamps, event order, before/after boundaries, retries, restart timing, and compaction timing. | Causality depends on order, latency, and recovery timing. | OpenHands conversation and event logs; OpenHands condenser insertion boundaries; OpenClaw trajectory timelines and compaction events. | Summaries can collapse ordered ranges into unordered prose, and visible replay timing may not equal decision timing. | Are all causal boundaries timestamped and linked to original event IDs? | inferred |
| contradiction reconstruction | Reconstructing conflicting evidence, stale assumptions, reversals, rejected alternatives, and uncertainty markers. | Failure explanation can depend on whether a system acted on a stale, contradicted, or unresolved premise. | Public compression, memory, transcript, and event surfaces may expose some conflict markers. | Summaries can flatten contradictions into resolved statements, and replay artifacts may not retain rejected branches. | Do public artifacts preserve enough contradiction lineage to distinguish resolved from unresolved context? | hypothesis |
| causal reconstruction | Reconstructing the chain from model-visible context, memory, tool results, environment state, configuration, and timing to later action. | This is the central adequacy question: visible replay is useful only if it can support bounded causal explanation. | Combined OpenClaw trajectory, memory, and compaction sources; combined OpenHands event, conversation, condenser, browser replay, and runtime sources. | Current public evidence establishes relevant surfaces, not complete causal sufficiency. | What minimum lineage bundle is enough to explain why a long-horizon failure path occurred? | unresolved |
| recovery-loop reconstruction | Reconstructing retries, fallback behavior, stuck detection, aborts, restarts, resumptions, cleanup, and state restoration. | Long-horizon failures often emerge from recovery loops, not only from the first error. | OpenClaw compaction and trajectory surfaces; OpenHands conversation status, runtime, and condensation surfaces. | Recovery may be partially visible while cleanup actions, fallback choices, or state restoration details remain implicit. | Are restart and recovery events connected to prior state, new state, and omitted ranges? | unresolved |
| intent reconstruction | Reconstructing user goals, constraints, approvals, prohibitions, changed scope, and task authority over time. | Operational failure explanation can depend on whether later action still matched the current authorized intent. | Transcript, memory, compaction, conversation, and event surfaces. | Summaries and memory recalls can simplify intent, lose source authority, or retain expired constraints. | Can public artifacts distinguish original intent, revised intent, summary-derived intent, and inferred intent? | hypothesis |
| replay provenance reconstruction | Reconstructing where replay data came from, what was captured, what was omitted, source hashes, and replay artifact lineage. | An audit must know whether the replay itself is complete enough for the reconstruction question. | Evidence manifests, trajectory bundle metadata, browser replay metadata, event IDs, omitted-event references, and source hashes. | Replay artifacts can be viewable while their capture method, omitted ranges, or source integrity remain incomplete. | Which provenance metadata is required before treating a replay as reconstruction evidence? | unresolved |
